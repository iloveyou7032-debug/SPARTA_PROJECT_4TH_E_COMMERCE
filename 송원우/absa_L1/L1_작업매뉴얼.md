## 1. Phase 개요

```
Phase A. Cross-Overlap 라벨링 (60~90분)
   ↓
Phase B. Gate 1: 라벨러간 일치도 검증 (10분)
   ├─ 통과 (κ ≥ 0.65) → Phase C 진입
   └─ 실패 → 가이드라인 v2.3 patch 후 송원우와 합의 미팅
   ↓
Phase C. absa.py 구현 (4~6시간)
   ↓
Phase D. Gate 2: 모델 검증 (30분)
   ├─ 통과 (Macro-F1 ≥ 0.70) → Phase E 진입
   └─ 실패 → Few-shot 보강 / 트리거 보강 → 재실행
   ↓
Phase E. 전체 데이터 추론 (3~6시간)
```

## 2. Phase A — Cross-Overlap 라벨링

### 2-1. 입력 파일

| 파일 | 위치 | 비고 |
|------|------|------|
| 라벨링 대상 | `송원우/final_data/absa_overlap_validation_L1.xlsx` | 시트 [1_검증라벨링] 사용 |
| 가이드라인 v2.2 | `송원우/absa_guideline_v2.md` | 라벨링 중 항상 참조 |
| 정답지 (라벨링 후 사용) | `송원우/final_data/_overlap_answer_key_v23_DO_NOT_OPEN_BEFORE_LABELING.xlsx` | **Phase B에서만 열기** |

### 2-2. 작업 절차

1. `absa_overlap_validation_L1.xlsx` 시트 [0_안내] 정독
2. 시트 [2_가이드라인_요약] 인쇄 또는 옆 화면에 띄움
3. 시트 [1_검증라벨링]에서 100건을 위에서부터 순차 라벨링
4. 각 행에 `기능성_라벨러1`, `브랜드_라벨러1` 입력 (드롭다운 P/N/X)
5. 확신 없는 케이스 → X + 메모 컬럼에 사유 기재
6. **정답지 절대 미열람** (Phase B에서만 사용)
7. 완료 후 `absa_overlap_validation_L1_DONE.xlsx`로 저장

### 2-3. 라벨링 핵심 원칙 (시트 [2_가이드라인_요약]에서 발췌)

- **기능성 착용감 귀속**: 의류=핏/사이즈(X), 신발/양말=기능성(P/N), 끈/밴드 강성=소재(X)
- **브랜드 협소화**: 재구매·추천·팬 명시만 P. "디자인 예쁨/핏 좋음/포장 좋음" = X (제품 만족 ≠ 브랜드 긍정)
- **확신 없으면 X**

## 4. Phase C — absa.py 구현

### 4-1. 골격 파일

`송원우/absa.py` — 송원우 작성 함수 시그니처+주석. **[L1 채우기]** 표시된 함수 본문을 구현.

### 4-2. 구현 순서 (권고)

| 순서 | 함수 | 주요 작업 |
|------|------|-----------|
| 1 | `load_golden_set()` | v2.3 로드 + 컬럼 검증 |
| 2 | `build_few_shot_examples()` | 속성별 P/N/X 각 3건 추출 (캘리브레이션 합의 우선) |
| 3 | `build_prompt()` | EXAONE Few-shot 프롬프트 (가이드라인 §3.{속성} 트리거 inject) |
| 4 | `call_exaone()` | ollama Python client (temperature=0.1, max_tokens=5) |
| 5 | `stage1_trigger_match()` | 트리거 사전 매칭 (absa_relabel.py 재활용) |
| 6 | `stage2_exaone()` | Stage1 매칭 속성만 EXAONE 호출 (병렬) |
| 7 | `predict_full_pipeline()` | A→1→2 통합 |
| 8 | `evaluate()` | Macro-F1, 속성별 F1, Confusion Matrix |

### 4-3. 캐스케이드 전략 (Option 3: Rule + LLM)

```
[리뷰 1건]
   ↓
Stage 1: 트리거 사전 매칭
   - 6속성 각각에 대해 P/N 트리거 키워드 매칭
   - 매칭 없음 → X 즉시 결정 (LLM 호출 안 함, 비용 절감)
   - 매칭 있음 → Stage 2로
   ↓
Stage 2: EXAONE Few-shot (모호 케이스만)
   - 트리거 P+N 동시 매칭 (혼재) → LLM이 더 강한 신호 선택
   - 트리거 단방향 매칭 → LLM이 맥락 검증 (false trigger 방지)
   - 결과: P / N / X
```

**예상 호출 수**: 1.16M 리뷰 × 평균 1.5속성(트리거 매칭) ≈ 1.7M LLM 호출 (Stage1만으로 ~70% 처리)

### 4-4. Few-shot 프롬프트 템플릿 (예시)

```
당신은 한국어 애슬레저 리뷰의 ABSA(Aspect-Based Sentiment Analysis) 전문가입니다.
다음 리뷰의 [{aspect}] 속성에 대한 감성을 P / N / X로 분류하세요.

규칙:
- P: {aspect_p_triggers}
- N: {aspect_n_triggers}
- X: 해당 속성 언급 없음 또는 판단 불가
{aspect_special_rules}

예시:
리뷰: "{example_1_content}"
답: {example_1_label}
...
리뷰: "{example_5_content}"
답: {example_5_label}

리뷰: "{target_content}"
답:
```

### 4-5. 골든셋 분할 (학습/검증)

- **Few-shot pool**: 속성별 P/N/X 각 5건 = 90건
- **Validation set**: 나머지 910건 → Macro-F1 측정용

## 5. Phase D — Gate 2: 모델 검증

### 5-1. 실행

`L1_실행노트북.py`의 **[Phase D]** 셀 실행.

### 5-2. 검증 지표

| 지표 | 목표 | 미달 시 조치 |
|------|------|------|
| Macro-F1 (6속성 평균) | ≥ 0.70 | Few-shot 5건→10건 보강, 트리거 사전 보강 |
| 속성별 F1 | ≥ 0.60 | 미달 속성만 Few-shot 보강 |
| 브랜드/헤리티지 F1 | ≥ 0.55 | 클래스 imbalance(P=202) → 가중 prompting |
| Confusion P↔N | < 5% | 부정 톤 룰 검토 |

### 5-3. 향상 절차 (반복)

```
1. 속성별 F1 산출 → 가장 낮은 속성 확인
2. 해당 속성 confusion matrix 분석
3. 오분류 패턴 → Few-shot 예시 추가 (5→10건)
4. 또는 트리거 사전 보강 (오분류 케이스의 키워드)
5. 재추론 → F1 재측정 (max 3회 반복)
```

### 5-4. Gate 2 실패 시 절차

- 3회 반복 후에도 미달 → 송원우에게 공유
- Stage1 트리거 사전 자체의 한계 가능성 → 송원우와 사전 보강 협업

## 9. 트러블슈팅

| 증상 | 원인 | 해결 |
|------|------|------|
| Ollama timeout | 동시 요청 과다 | 병렬 수 4→2로 축소 |
| EXAONE 응답 'P/N/X' 외 | 프롬프트 출력 형식 명시 부족 | "답은 P, N, X 중 하나만 출력" 추가 |
| F1 낮은 속성 한 개만 부진 | Few-shot 다양성 부족 | 해당 속성 P/N/X 각 5건 추가 |
| 메모리 부족 (Phase E) | 한 번에 전체 로드 | chunksize=10000 사용 |
| 프로세스 중단 | 장시간 실행 | 매 100K parquet append → 마지막 idx 이후 재개 |
