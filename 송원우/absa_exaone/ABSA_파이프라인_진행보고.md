# ABSA 파이프라인 진행 보고
> 작성: 안진식 | 최종 업데이트: 2026-05-08

---

## 1. 전체 목적 및 흐름

### 왜 하는가?

애슬레저 브랜드(안다르 / 젝시믹스 / 휠라 / 룰루레몬) 리뷰 **116만 건**에서  
단순 별점이 아닌 **속성별 감성(P/N/X)** 을 추출하기 위함.

> 예) "소재는 좋은데 사이즈가 작아요" → 소재=P, 핏/사이즈=N, 나머지=X

이를 통해 브랜드별 강점/약점, 개선 포인트를 속성 단위로 비교 분석.

### 분석 대상 6개 속성

| 속성 | 예시 표현 |
|------|----------|
| 핏/사이즈 | 정사이즈, 사이즈 작음, Y존 커버 |
| 소재/내구성 | 부드럽다, 보풀, 변색 |
| 기능성 | 신축성, 땀 흡수, 통기성 |
| 디자인 | 예쁘다, 색감, 실루엣 |
| 브랜드/헤리티지 | 재구매, 강추, 실망 |
| 가격/가치 | 가성비, 너무 비쌈 |

---

## 2. 전체 진행 단계 (Phase)

```
Phase A  라벨링 (골든셋 1,000건 — 6속성 × P/N/X 직접 라벨링)
    ↓
Phase B  Gate 1: 라벨러 간 일치도 검증 (Cohen's κ ≥ 0.65)
         결과: 기능성 κ=0.6789 / 브랜드 κ=0.7493 → Macro κ=0.7141 ✅ 통과
    ↓
Phase C  Few-shot 구성 + 골든셋 분할 (검증셋 오염 방지)
    ↓
Phase D  Gate 2: 모델 자동 분류 검증 (Macro F1 ≥ 0.70)
         → v1~v4 반복 개선 (아래 참고)
    ↓
Phase E  층화 샘플링 (브랜드당 2,000건 × 4 = 8,000건)
         NPS 기준: 별점 0~3 → negative / 4~5 → positive
    ↓
Phase F  전체 ABSA 추론 → 분석 / 시각화
```

## 3. Phase별 상세 설명

### Phase A — 골든셋 라벨링

골든셋 1,000건을 직접 읽으며 6개 속성마다 P/N/X를 붙이는 작업.  
이 라벨이 모델 평가의 **정답지(gold label)** 가 됨.

### Phase B — Gate 1: 라벨러 간 일치도 (Cohen's κ)

같은 리뷰를 두 사람(안진식, 송원우)이 각자 라벨링하고 일치도를 측정.  
**왜 하는가?** 라벨이 사람마다 달라지면 정답지 자체가 신뢰할 수 없어 모델 평가가 의미 없어짐.  
κ ≥ 0.65 = "Substantial Agreement" 기준 통과해야 Phase C 진입 가능.

결과: Macro κ = **0.7141** ✅ 통과

### Phase C — Few-shot 구성 및 골든셋 분할

**Few-shot이란?**  
LLM에게 정답 예시를 미리 보여주는 것. 예를 들어:
```
리뷰: "원단이 너무 부드러워요"  → 소재/내구성: P
리뷰: "사이즈가 너무 작아요"    → 핏/사이즈: N
리뷰: "배송 빠르네요"           → 핏/사이즈: X (언급 없음)
```
이런 예시를 프롬프트에 넣어주면 LLM이 판단 기준을 더 잘 이해함.

**어떻게 구성했나?**
- 골든셋 1,000건에서 속성별로 P/N/X 각 3건씩 = 최대 54건 후보 선택
- 실제 few-shot으로 사용된 건수: **27건** (중복 리뷰 제외)

**왜 분리하는가? (골든셋 분할)**  
few-shot으로 쓴 27건을 검증셋에도 포함하면 **데이터 누수(data leakage)** 발생.  
모델이 "본 적 있는 리뷰"를 맞히는 건 실력이 아니므로, few-shot 27건은 검증에서 제외.

```
골든셋 1,000건
├── Few-shot용:  27건 → LLM 프롬프트 예시로 사용
└── 검증셋:     973건 → 모델 성능 측정 (정답 모르는 상태로 예측)
```

### Phase D — Gate 2: 모델 성능 검증 (Macro F1)

검증셋 973건에 대해 모델이 자동 예측한 결과를 골드 라벨과 비교.  
**Macro F1** = 6개 속성의 F1 점수 평균. 목표: **≥ 0.70**

F1 점수란?
- Precision(정밀도): 모델이 P라고 한 것 중 실제 P의 비율
- Recall(재현율): 실제 P 중 모델이 P라고 맞힌 비율
- F1 = 둘의 조화 평균 → 하나만 높아도 안 되고 둘 다 높아야 함

### v1 — absa.py (Macro F1: **0.4162** ❌)

**구조 설명**

리뷰 1건을 처리할 때 두 단계를 거쳤습니다.

- **Stage 1 (트리거 매칭)**: 먼저 키워드 목록(`TRIGGERS`)으로 빠르게 스캔.  
  예: "재구매"라는 단어가 있으면 → 브랜드/헤리티지 = P로 바로 확정.  
  키워드가 없으면 → X(언급 없음)로 판정하고 Stage 2 **스킵**.

- **Stage 2 (LLM 개별 호출)**: Stage 1에서 X가 아닌 속성만 LLM에 질문.  
  단, 6개 속성을 **하나씩 따로** 호출 → 리뷰 1건당 최대 6번 API 호출.  
  (28초/건이 걸린 이유)

**핵심 버그**: Stage 1이 X로 판정하면 LLM을 아예 호출하지 않음

```python
if label == 'X':
    out[aspect] = 'X'  # LLM 미호출 → 그냥 X로 끝
```

문제는 트리거 키워드가 한국어 불규칙 활용을 커버하지 못했다는 점입니다.

```
트리거에 '예쁘' 등록
→ "예쁘게" ✅ 매칭
→ "예뻐요" ❌ 미매칭  ← 예쁘 + 아/어 → 예뻐 (불규칙 변화)
→ "예쁜데" ❌ 미매칭
```

결과적으로 85~99%의 리뷰가 Stage 1에서 X로 떨어져 LLM 없이 X로 출력됨  
→ P/N recall이 0.04~0.24로 붕괴

### v2 — absa_v2.py (Macro F1: **0.6474** ❌)

**변경**: Stage 1 X 숏컷 제거 + 6속성 **배치 1회 호출**로 개선

v1의 핵심 버그를 제거하고, 속성별 6번 따로 묻던 것을 **한 번의 프롬프트**에  
6속성을 모두 담아 LLM에 한꺼번에 질문하도록 변경.

```
Before (v1): [핏/사이즈?] → [소재?] → [기능성?] → ... × 6번 = 28초
After  (v2): [핏/사이즈, 소재, 기능성, 디자인, 브랜드, 가격 한번에?] = 7.7초
```

**남은 문제**: 브랜드/헤리티지 F1 = 0.4021  
→ 973건 중 931건(95.7%)을 X로 예측 (gold X는 672건/69%)  
→ 배치 프롬프트에 브랜드 판단 기준이 없어 불확실하면 X로 도망

**실패 원인 분석**

브랜드/헤리티지가 Macro F1을 0.65 아래로 끌어내리는 주범.

```
[브랜드/헤리티지] 오분류 282건
  gold=P → pred=X  (183건)  ← P 리뷰의 92%를 X로 놓침
  gold=N → pred=X  ( 87건)  ← N 리뷰도 X로 도망
  gold=X → pred=P  (  7건)  (미미)
```

원인: 배치 프롬프트에 브랜드 판단 기준이 없어 LLM이 불확실하면 X로 도망.  
"재구매했어요", "강추해요" 같은 명확한 브랜드 표현도 X로 출력.

소재/내구성도 P→X(99건), N→X(101건)으로 X 편향이 심했으나,  
브랜드보다 리뷰당 언급 빈도가 높아 자연 보정됨.

**대응**: `absa_v3.py` — 브랜드 판단 기준 프롬프트 명시 + 트리거 교체

### v3 — absa_v3.py (Macro F1: **0.6534** ❌)

**변경 3가지**:

1. **TRIGGERS를 P/N 분리 구조로 전면 교체** (absa_relabel.py v2.2 기준)  
   기존: 트리거 하나로 P/N 구분 없이 사용  
   변경: TRIGGERS_P(긍정 키워드), TRIGGERS_N(부정 키워드)로 분리  
   브랜드/헤리티지 키워드 대폭 보강 (재구매·추천 → 브랜드명·감성어 추가)

2. **배치 프롬프트에 브랜드/헤리티지 판단 기준 명시 + X 남용 경고**  
   LLM에게 "브랜드 관련 표현이면 X가 아닌 P/N을 써라"고 명시

3. **`apply_trigger_correction()` 추가**  
   LLM이 X로 예측했더라도, 트리거 키워드가 매칭되면 P/N으로 덮어쓰는 후보정 로직

**결과**: 브랜드/헤리티지 F1 0.40 → **0.56** 으로 개선  
**새 문제**: 핏/사이즈 F1이 0.67 → **0.58로 하락** (트리거 과잉 매칭 부작용)

**실패 원인 분석**

v2 대비 브랜드는 개선됐지만 핏/사이즈·브랜드 두 속성이 동시에 문제.

```
[브랜드/헤리티지] 오분류 340건
  gold=X → pred=P  (153건)  ← v2: 7건 → v3: 153건 (트리거 추가 후 급증)
  gold=P → pred=X  (108건)  ← P를 여전히 X로 놓침
  gold=N → pred=X  ( 41건)

[핏/사이즈] 오분류 382건
  gold=X → pred=P  (194건)  ← 핏 무관 리뷰를 P로 오분류
  gold=X → pred=N  (133건)  ← 핏 무관 리뷰를 N으로 오분류
  gold=P → pred=N  ( 28건)
```

핏/사이즈: 트리거를 TRIGGERS_P/N으로 분리 교체하면서 매칭 범위가 확대됨.  
"무난하게 입기 좋아요" 같이 핏 직접 언급이 없는 리뷰도 트리거 히트 → X 골드를 P/N으로 오보정.

브랜드: 트리거 추가로 X→P 오보정이 7건 → 153건으로 급증.  
동시에 P→X(108건)도 여전히 높아, 양방향 오류가 공존.

**대응**: `absa_v4.py` — KiWi 형태소 분석기 통합으로 트리거 정밀도 향상 시도

### v4 — absa_v4.py (진행 중)

**v3 문제 근본 원인 해결**: 트리거가 한국어 불규칙 활용을 여전히 못 잡는 문제  
→ 송원우 팀원의 `preprocess.py` KiWi 형태소 분석기를 통합해 해결

**변경 4가지**:

1. **KiWi 형태소 토크나이저 통합**  
   리뷰 원문을 형태소 분석 후 어간 정규화 → 불규칙 활용 자동 처리
   ```
   "예뻐요"    → tokens: "예쁘다"   ← 트리거 '예쁘' 매칭 ✅
   "부드러워요" → tokens: "부드럽다" ← 트리거 '부드럽' 매칭 ✅
   "이쁘다"    → tokens: "예쁘다"   ← NORMALIZATION_DICT 정규화 ✅
   ```

2. **트리거 매칭은 tokens(형태소) / LLM 프롬프트는 원문 분리**  
   LLM에는 자연스러운 원문을 주고, 키워드 매칭만 정규화된 tokens로 수행

3. **Stage 1 확정 속성은 LLM 스킵** → 속도 향상  
   트리거가 P 또는 N으로 명확히 확정한 속성은 LLM 호출에서 제외

4. **배치 예시 3건 → 6건** + **ThreadPoolExecutor(workers=2) 병렬화**  
   GPU 활용률 개선 (RTX 3050 4GB 기준 workers=2 최적)

**실패 원인 분석**

v4에서 새로 추가한 "Stage 1 확정 속성 → LLM 스킵" 로직이 역효과.

KiWi 형태소 정규화로 트리거 매칭이 더 많이 되는 것은 좋으나,  
잘못 매칭된 케이스도 LLM 검증 없이 그대로 확정 → **false positive 급증**.

## 6. Phase E — 층화 샘플링 + 전체 추론

### Phase E-1. 속성 다양성 기반 층화 샘플링 (셀: `958e617e`)

**목적**: 전체 116만 건 중 ABSA 추론할 대상 리뷰를 브랜드·감성·속성 균형 있게 선별

**샘플링 기준 (1:10 비율)**

| 구분 | 건수 | 근거 |
|------|------|------|
| 부정 (별점 1~3★) | 274건/브랜드 | FILA 부정 전수(병목) |
| 긍정 (별점 4~5★) | 2,740건/브랜드 | 274 × 10 |
| 브랜드당 합계 | 3,014건 | |
| **4브랜드 총합** | **12,056건** | 예상 추론 ~29시간 |

**왜 1:10인가?**
- 통계적 타당성: 브랜드 간 비율을 동일하게 유지해야 비교 가능
- FILA 부정 274건이 병목 → 모든 브랜드 부정을 274건으로 통일
- 긍정은 풀이 충분히 크므로 10배수로 확장 → 속성별 세분화 분석 신뢰도 확보

**속성 태깅 방식**
- v9의 `TRIGGERS_N` / `TRIGGERS_P` (속성별 키워드 딕셔너리) 활용
- `tag_aspects(text, triggers)`: 매칭된 속성 리스트 반환 (복수 매칭 가능)
- 결과 컬럼: `neg_aspects`, `pos_aspects` (예: `"핏/사이즈, 소재/내구성"`)

**속성 균등 샘플링 — 희소 속성 우선 배정 (exclusive assignment)**

일반 균등 샘플링 문제: 복합 속성 리뷰(예: 핏+소재 동시 언급)가 여러 속성 카운트에 중복 계산 → 특정 속성 쏠림

해결: **희소 속성 우선 배정**
```
1. 6개 속성을 풀 크기 오름차순 정렬 (희소 속성 먼저)
2. 각 속성마다 아직 미선택된 리뷰 중 n_per_asp건 선택
   → 한 리뷰는 한 속성에만 배정 (exclusive)
3. 부족분은 태깅 무관 랜덤 보충
```

결과: 안다르·젝시믹스는 속성당 약 45건 균등, FILA·룰루레몬은 부정 풀 소규모로 태깅 없음 비율 높음 (FILA 75%, 룰루레몬 53% — TRIGGERS 키워드 커버리지 한계, ABSA 추론 자체에는 영향 없음)

**출력 파일**: `absa_phase_e_sample.parquet`

**분포 확인 셀** (`d4fb3f5a`): E-1 직후 실행. 샘플 구성 적합성 검증.

```
absa_phase_e_sample.parquet 로드 (12,056건)
```

**예상 소요시간**: 12,056건 × 8.62초 ÷ 2(workers) ≈ **약 29시간**

**Windows 절전 방지 (ctypes)**

장시간 추론 중 화면 끄기·절전·하이버네이션으로 Ollama가 중단되는 것 방지.
```python
ES_CONTINUOUS | ES_SYSTEM_REQUIRED | ES_DISPLAY_REQUIRED
SetThreadExecutionState(...)  # 추론 시작 시 ON
# try/finally로 완료/에러 시 항상 해제
SetThreadExecutionState(ES_CONTINUOUS)  # OFF
```

**체크포인트 복구 방식**

E-2를 중간에 인터럽트하더라도 `absa_phase_e_checkpoint.csv`가 남아있으면 재실행 시 완료된 건 수 건너뛰고 이어받기. 단, 이전 샘플 파일(`absa_phase_e_sample.parquet`)이 동일해야 인덱스가 일치.
